# cir2c — SV-COMP reachability evaluation harness

This directory evaluates the **cir2c** C/C++→C mapping by running SV-COMP
reachability verifiers over three task suites and comparing the results:

| suite | what it is |
|-------|------------|
| `esbmc-eval` | the original ESBMC-eval benchmarks (C++) |
| `esbmc-eval-mapped` | cir2c output, `--externalize-std` (default) |
| `esbmc-eval-mapped-nostd` | cir2c output, `--no-externalize-std` |

The question is whether mapping C++ to verifier-friendly C lets C-only verifiers
handle this suite, and at what cost — so the original-C++–capable verifiers
(**esbmc-kind, cbmc, divine**) run all three suites, while every other verifier
runs only the two mapped suites and is tabled against the ESBMC baseline.

Everything is data-driven: the tool set comes from SV-COMP's category structure,
per-tool details from fm-tools, and the Makefile auto-detects whatever is in
`xml/`. The only hardcoded fact is the three original-C++ verifiers above.

## Layout

| path | role |
|------|------|
| `Makefile` | orchestrator: download, run, normalize, tables (auto-detects `xml/*.xml`) |
| `gen-xml.py` | generates `xml/<tool>.xml`, one per C.ReachSafety verifier |
| `_fmtools.py` | shared fm-tools/category accessors (tool set, versions, options, URLs, auto-clone) |
| `fmtools-url.py` | prints a tool's download URL (used by the Makefile) |
| `setup-sources.py` | populates `sources/` with the three suites (runs cir2c) |
| `normalize-results.py` | strips the suite prefix from result names so columns join |
| `category-structure.yml` | vendored SV-COMP category file — defines the tool set |
| `xml/` | generated benchmark definitions (drop one to disable that tool) |
| `sets-full/`, `sets-smoke/` | task sets; `sets` is a symlink to one of them (see `SETS`) |
| `sources/` | the three task suites + `unreach-call.prp` (generated by `setup-sources.py`) |
| `tools/fm-tools/` | SV-COMP tool database — gitignored, cloned on demand |
| `tools/bin/<tool>/` | downloaded verifier archives |
| `results/`, `tables/` | run outputs and generated comparison tables (gitignored) |

## Prerequisites

- `python3` with `PyYAML`; `git`, `curl`, `unzip` on `PATH`.
- BenchExec is bundled under `benchexec/`. Runs use **local BenchExec** by default;
  set `RUNNER=vcloud` to dispatch to the **VerifierCloud**
  (`benchexec/contrib/vcloud-benchmark.py`) instead, which needs VerifierCloud access.
- fm-tools is **not** required up front: it is cloned automatically the first
  time a download or `gen-xml.py` needs it.

## Workflow

### 0. Generate the task suites (once)

```bash
python3 setup-sources.py            # populate sources/ — slow: runs cir2c per task
python3 setup-sources.py -n         # dry run (print actions only)
python3 setup-sources.py --yaml-only   # fast: only (re)write task .yml files
```

`--yaml-only` is also the quick way to migrate an existing checkout to the
`unreach-call` property name without re-mapping.

### 1. Choose a task set

`SETS` has no default, so the choice is always explicit. It links `sets` →
`sets-full/` or `sets-smoke/`:

- `SETS=smoke` — 10 tasks per suite, for validating the pipeline end-to-end.
- `SETS=full` — the whole suite (1532 / 1446 / 1446 tasks).

### 2. Run verifiers

```bash
make SETS=smoke esbmc-kind        # one tool: clone fm-tools + download tool + run
make SETS=full all -j8            # every detected tool, in parallel
make download-all                 # just fetch every tool archive
make list                         # show the detected tool split
```

A tool's archive is fetched on first use into `tools/bin/<tool>/` (single
top-level folder in the zip → renamed to the tool; otherwise boxed into a new
tool-named folder).

### 3. Build tables

```bash
make tables                       # normalize results, then generate tables/
```

Produces, under `tables/`:

- `<tool>-all` — for each of esbmc-kind / cbmc / divine: its three suites side by side.
- `esbmc-vs-<tool>` — for every other tool: ESBMC (original) vs the tool on the
  mapped and mapped-nostd suites.

## Command reference

| command | effect |
|---------|--------|
| `make list` | print all / three-config / two-config tool sets |
| `make SETS=smoke\|full <tool>` | download (if needed) and run one tool |
| `make SETS=smoke\|full all [-jN]` | run every detected tool |
| `make download-all` | download every tool archive |
| `make tools/fm-tools` | clone the fm-tools database |
| `make normalize` | normalize `results/*.xml.bz2` into `results/normalized/` |
| `make tables` | normalize + build comparison tables |

## Customizing

- **Runner** — `RUNNER ?= benchexec` (local) or `RUNNER=vcloud` (VerifierCloud).
  vcloud-only flags (priority, client heap, ivy cache) are added only for `vcloud`.
- **Resource budget** — SV-COMP defaults: CPU time `TIMEOUT ?= 900s` (Makefile),
  memory `15 GB` and `8` cores (`MEMLIMIT` / `CPU_CORES` in `gen-xml.py`; regenerate
  after changing).
- **Tool set** — driven by `category-structure.yml`'s `C.ReachSafety.verifiers`.
  To disable a tool just delete its `xml/<tool>.xml`. To re-derive the set (e.g.
  after a new SV-COMP edition): refresh the category file (command in
  `_fmtools.py`) and run `python3 gen-xml.py --clean`.
- **Versions** — `TARGET_COMPETITION` in `_fmtools.py` selects which SV-COMP
  edition's archives/options each tool uses (kept at the last edition that ships
  archives for every ReachSafety tool).
- **Baseline** — `BASELINE ?= esbmc-kind` in the Makefile anchors the
  `esbmc-vs-<tool>` tables.

## Notes

- The verification property is the SV-COMP standard `unreach-call`
  (`properties/unreach-call.prp`): `reach_error()` is never reachable.
- `xml/*.xml` are generated — edit `gen-xml.py` (or fm-tools/category data) and
  regenerate rather than editing them by hand.
- Each run is launched from inside its `tools/bin/<tool>/` directory, so the
  verifier binary is found without `--tool-directory` (and under `vcloud`, only that
  tool is shipped — not the gitignored `tools/fm-tools` clone). `-o` keeps results
  in `results/` regardless of the cwd.
